Security Verification

Verify our security claims using your browser's developer tools

Step-by-Step Security Verification

Test 1 Network Traffic Monitoring

Objective: Verify that credentials are not sent to our server during form submission

  1. Open your browser's Developer Tools (F12 or right-click ‚Üí Inspect)
  2. Go to the Network tab
  3. Clear any existing entries (üóëÔ∏è clear button)
  4. Navigate to our Setup Form
  5. Fill in the form with test credentials (use fake data)
  6. Submit the form
  7. Verify: Check all network requests - credentials should only appear in redirects (URL parameters), never in request bodies to our server

Expected Result: You should see POST to /setup contains only form data, but no subsequent storage requests to databases or external services.

Test 2 Local Storage Inspection

Objective: Confirm no credentials are stored in browser storage

  1. Open Developer Tools ‚Üí Application tab (Chrome) or Storage tab (Firefox)
  2. Check Local Storage for this domain
  3. Check Session Storage for this domain
  4. Check Cookies for this domain
  5. Fill out and submit the form with test data
  6. Re-check all storage locations

Expected Result: Only theme preference should be stored. No API keys, passwords, or personal data should appear in any storage.

Test 3 Console Error Monitoring

Objective: Verify application handles credentials securely without errors

  1. Open Developer Tools ‚Üí Console tab
  2. Clear the console
  3. Navigate through the entire application flow
  4. Submit forms with both valid and invalid test data
  5. Monitor for any credential leakage in console logs or error messages

Expected Result: No API keys or sensitive data should appear in console logs, even during error conditions.

Test 4 Client-Side Code Inspection

Objective: Review client-side code for security implementation

  1. Open Developer Tools ‚Üí Sources tab
  2. Examine JavaScript files served by the application
  3. Look for credential handling functions
  4. Verify credentials are only passed to legitimate API endpoints
  5. Check for any data collection or tracking code

Expected Result: Code should show credentials are immediately used for API calls to Mailgun/Cloudflare, with no storage or transmission to our servers.

Our Security Architecture

What We DO

  • Process credentials in-browser only
  • Send API calls directly to Mailgun/Cloudflare
  • Use HTTPS for all communications
  • Store only theme preferences locally

What We DON'T DO

  • Store credentials on our servers
  • Log or monitor your API usage
  • Use analytics or tracking cookies
  • Connect to external data collection services
